### 项目定位

Chaos Code 深受 Claude Code 和 Claw Code 的启发，是一个**专注于命令行交互、具备完全自主能力的 AI 协同开发框架**。它深度融合了：

- **规范驱动开发 (Spec-Driven Development)** — 强调先写规格、再写代码
- **测试驱动开发 (TDD)** — 强调先写测试、再写实现
- **Model Context Protocol (MCP)** — 多模型兼容，动态工具扩展

核心理念：让 AI 代理能够"自主读取代码、规划修改、运行本地测试、修复错误，并直接处理 Git 提交与推送"，形成完整的开发闭环。

## 2. 核心特性

### 2.1 自主交互与闭环 (Claude-like Shell)

- 直接在终端以交互式会话 (REPL) 运行
- AI 可以智能分析项目结构、自主检索文件、撰写统一补丁 (Unified Diff)、并自动执行本地测试
- 支持多轮对话，上下文感知，可持续追踪项目状态
- 流式 Markdown 渲染、Tab 三层补全（命令/模型/文件）、多行输入、动态 Prompt（git branch + model）
- 实时工具调用结构化展示、每轮计时/Token/费用摘要

### 2.2 完整的 Git 自动化链条

- AI 代理可自动调用 Git 写入命令：`git.add`、`git.commit`、`git.push`、`git.checkout`、`git.branch`、`git.reset`
- 提供会话内变更预览及安全确认机制
- 一键 `/rollback` 回滚、`/commit` 交互提交、`/undo` 恢复 AI 修改的文件

### 2.3 强大的 MCP 协议支持

- 内置高适配 Model Context Protocol 客户端
- 可动态拉起外部 stdio 协议 MCP 服务 (数据库操作、浏览器驱动等)
- 将外部工具动态反射至大模型认知网络中
- 通过 `stdd/mcp-servers.json` 配置 MCP 服务

### 2.4 智能自愈测试环 (Self-Healing Loop)

- 测试运行 (`test_run`) 失败时，AI 代理自动解析错误日志与调用栈
- 生成自愈提示并进入"修补→再测试"循环
- 直至全部测试验证通过

### 2.5 多模型自由切换与开销监控

- 兼容 OpenAI、Anthropic、DeepSeek、OpenRouter、Groq、Ollama 等提供商
- 交互终端中通过 `/model` 实时无缝切换模型
- 自动实时折算和统计会话 Token 开销与估算费用
- 支持 `--model` 和 `--provider` CLI 标志

### 2.6 21+ 内置工具 + MCP 动态扩展

- `fs_read`: 读取文件
- `fs_search`: 基础文本搜索
- `fs_glob`: Glob 模式文件查找（`**/*.js`、`src/**/*.ts`）
- `fs_grep`: 正则搜索 + 上下文行 + glob 过滤 + 三种输出模式
- `fs_write`: 直接写文件（创建/覆盖）
- `fs_patch`: 差异补丁
- `shell_run`: Shell 命令执行
- `test_run`: 测试运行
- 全套 Git 工具：`git_diff`、`git_add`、`git_commit`、`git_push`、`git_checkout`、`git_branch`、`git_reset`
- STDD 工具：`stdd_status`、`stdd_recommend`、`stdd_verify`
- 任务管理：`task_create`、`task_update`、`task_list`（创建/更新/列出任务及依赖关系）
- MCP 动态工具

### 2.7 持久化配置与会话管理

- `~/.chaos/config.json` 管理所有偏好（verbosity、压缩阈值、温度、最大轮次、per-tool 权限）
- `~/.chaos/sessions/` 自动保存聊天历史，支持 `/resume` 恢复
- `/export` 导出对话为 Markdown
- `/history` 搜索跨会话持久化命令历史
- 权限持久化：按 `s` 保存 always-allow 到配置

### 2.6 Ralph Loop 规范驱动流程

八阶段严谨流程：`inspect → propose → spec → plan → patch → test → verify → summarize`

### 2.7 丰富的模板与工具生态

- **91 个 CLI 命令文件**，覆盖从初始化到部署的全生命周期
- **86 个命令模板** + **53 个 Skill 模板**
- 内置 6 个官方扩展模块
- 支持 Go / Python / TypeScript / JavaScript / Rust 五种语言的项目模板

## 4. 系统架构分析

### 4.1 整体架构分层

Chaos Code 采用**三层架构**设计：

```
┌────────────────────────────────────────────────────┐
│                    CLI 入口层                        │
│  cli.js → Commander.js 路由 → 命令注册中心          │
│  (chaos-terminal.js 交互式 REPL / 单次 Prompt)      │
├────────────────────────────────────────────────────┤
│                   命令执行层                         │
│  91 个 CLI Command 类 + 29 个工具模块               │
│  (init / apply / verify / graph / constitution...)  │
├────────────────────────────────────────────────────┤
│                   代理内核层                         │
│  AgentKernel → 工具注册 → 权限策略 → 执行引擎       │
│  ChaosAgentLoop (LLM 调用 → 工具执行 → 自愈循环)    │
│  MCP Client Manager (外部工具动态注册)              │
└────────────────────────────────────────────────────┘
```

### 4.2 核心模块职责

#### CLI 入口 (`cli.js`)

- 使用 Commander.js 注册所有子命令
- 三种运行模式：
  - **无参数** → 启动交互式 REPL (`launchChaosTerminal`)
  - **已知子命令** → 正常解析执行 (`program.parse`)
  - **未知参数** → 作为一次性 Prompt 送给 AI 代理 (`runChaosAgentPrompt`)
- 通过 `CommandLoader` 动态加载和注册 91 个命令类
- 集成进度追踪 (`session-progress`)，自动记录命令执行状态

#### 交互式终端 (`chaos-terminal.js`)

- 基于 Node.js readline 的 REPL 界面
- 彩色 ASCII Art 欢迎横幅
- 支持 25 种斜杠命令 (`/help`, `/diff`, `/commit`, `/rollback`, `/undo`, `/config`, `/resume` 等)
- 与 `ChaosAgentLoop` 深度集成，支持多轮对话
- 写入操作前弹确认提示 + diff 预览 (Permission Gate)
- 流式 Markdown 渲染 + Tab 三层补全 + 多行输入 + 动态 Prompt
- 实时工具调用展示 + 每轮计时/Token/费用摘要
- 自动压缩 + 会话持久化 + 配置持久化

#### 代理内核 (`AgentKernel` - `index.js`)

核心调度引擎，管理所有工具的注册、权限判断与执行路由：

- **工具路由**：根据工具名分发到具体执行器
- **权限策略**：通过 `PermissionPolicy` 决定 allow/ask/deny
- **生命周期管理**：MCP 客户端启动、STDD 工具初始化
- **循环运行**：补丁循环 (`runPatchCycle`)、修复循环 (`runRepairCycle`)、LLM 修复 (`runLlmRepair`)

#### 自主控制环 (`ChaosAgentLoop`)

Claude-like 的核心循环，驱动 LLM 与工具的交互：

- **LLM 调用**：兼容 OpenAI 和 Anthropic API
- **工具执行**：18 种内置工具 + MCP 动态工具
- **自愈机制**：测试失败时自动注入修复提示，循环直到通过
- **Token 追踪**：实时统计 Prompt/Completion Token 及费用估算
- **上下文压缩**：`compactHistory` 智能截断历史消息节省 Token

### 4.3 核心调用关系

```
用户输入
  │
  ├─→ cli.js (路由判断)
  │     ├─ 无参数 → chaos-terminal.js (REPL)
  │     ├─ 已知命令 → CommandLoader → 具体 Command 类
  │     └─ 未知参数 → runChaosAgentPrompt()
  │
  ├─→ ChaosAgentLoop.run(goal, history, callbacks)
  │     │
  │     ├─→ _callLLM(messages) ──→ OpenAI / Anthropic API
  │     │     │
  │     │     └─→ 返回 content + toolCalls
  │     │
  │     ├─→ _executeTool(name, args) ──→ AgentKernel.executeTool()
  │     │     │
  │     │     ├─→ PermissionPolicy.decide() (权限判断)
  │     │     ├─→ fs.read / fs.search → ReadOnlyToolExecutor
  │     │     ├─→ fs.patch → PatchTool
  │     │     ├─→ test.run → TestTool
  │     │     ├─→ shell.run → ShellTool
  │     │     ├─→ git.* → GitTool
  │     │     ├─→ stdd.* → StddTools
  │     │     └─→ mcp_* → McpClientManager.callTool()
  │     │
  │     └─→ 自愈检测: test_run 失败 → 注入修复提示 → 继续循环
  │
  └─→ AgentKernel 工具执行流程
        ├─→ ToolRegistry (工具注册与查找)
        ├─→ PermissionPolicy (权限决策: allow/ask/deny)
        ├─→ AgentSessionTrace (会话追踪日志)
        └─→ 具体工具实现 (GitTool / PatchTool / TestTool...)
```

### 4.4 数据流

```
[用户请求] → [REPL/CLI] → [ChaosAgentLoop]
                              │
                              ├─→ [LLM API] → 返回文本 + 工具调用意图
                              │
                              ├─→ [AgentKernel] → 权限检查 → 工具执行
                              │     │
                              │     ├─→ fs.read/search/glob/grep → 读取项目文件
                              │     ├─→ fs.write → 创建或覆盖文件
                              │     ├─→ fs.patch → 写入代码变更
                              │     ├─→ test.run → 运行测试
                              │     ├─→ git.* → Git 版本控制操作
                              │     ├─→ shell.run → 执行 Shell 命令
                              │     └─→ mcp_* → 调用外部 MCP 服务
                              │
                              └─→ [自愈循环] → 测试失败 → 自动修复 → 重测
                                    │
                                    └─→ [SessionTrace] → 记录全链路日志
```

## 11. 权限与安全模型

### 11.1 三级权限策略

| 级别 | 说明 |
|------|------|
| `allow` | 直接允许执行 (只读工具) |
| `ask` | 需用户确认 (写入工具) |
| `deny` | 策略拒绝 |

### 11.2 工具风险分级

| 风险等级 | 说明 | 示例 |
|---------|------|------|
| `safe` | 只读操作，无副作用 | `fs.read`, `fs.search`, `fs.glob`, `fs.grep`, `git.diff`, `stdd.status` |
| `guarded` | 受限操作，需策略检查 | `test.run`, `stdd.verify` |
| `write` | 写入操作，需用户确认 | `fs.write`, `fs.patch`, `shell.run`, `git.commit`, `mcp_*` |

### 11.3 安全机制

- 写入工具执行前弹出交互确认 (`askPermission` 回调)
- Shell 命令执行有超时限制
- Git 操作有安全边界检查
- Constitution Article 7: Security by Design
- `npm audit --audit-level=high` 安全审计
